# 候选人任务：天气应用

这是一份简短的现场编程考核，面向 Vue 前端和本地后端。

## 时间安排

- 考核前 15 分钟：安装依赖并启动项目
- 60 分钟：完成编程任务
- 不要求进行大型 UI 重构；简单、可用的界面即可
- 本次考核更强调后端能力

## 你的任务

请只选择一条路线。

### 路线 A

完成：

- 1 个前端任务
- 2 个后端任务

### 路线 B

完成：

- 1 个全栈任务
- 1 个后端任务

重要：不要把原始练习和这些任务一起完成。只需完成路线 A 或路线 B 下所选的任务。现有的城市天气查询功能只需要继续正常工作即可。

## 前端

选择 1 个。

### FE1：验证与反馈

- 去掉城市输入前后的空格
- 阻止空搜索
- 显示加载和错误状态
- 防止重复提交
- 区分“城市未找到”和“后端失败”

### FE2：防抖搜索

- 用户停止输入后再搜索
- 避免空值和不必要的请求
- 防止旧响应覆盖较新的响应

### FE3：最近搜索记录

- 使用浏览器存储
- 仅保存成功的搜索
- 最多保留五条记录
- 去掉大小写不敏感的重复项
- 允许再次搜索历史城市
- 安全处理损坏的存储数据

## 后端

路线 A 选择 2 个，路线 B 选择 1 个。

### BE1：验证与错误处理

- 验证并规范化城市输入
- 拒绝空输入
- 使用合适的状态码
- 使用一致的错误响应
- 不向前端暴露原始异常

### BE2：外部 API 可靠性

- 为 Open-Meteo 请求添加超时
- 处理连接失败和超时
- 处理上游数据缺失或格式错误
- 添加有用的日志
- 返回安全的前端错误信息

### BE3：内存缓存

- 缓存成功的天气结果
- 规范化缓存键
- 使用可配置过期时间
- 不缓存失败请求
- 允许候选人演示缓存命中和缓存未命中

## 全栈

路线 B 选择 1 个。前端必须只调用本地后端。

### FS1：空气质量

使用 Open-Meteo Air Quality API 并显示：

- AQI，并明确标注 AQI 标准
- PM2.5
- PM10

使用后端已经解析出的坐标。如果空气质量数据失败，天气结果仍应显示。

### FS2：多个同名城市

- 后端返回最多五个匹配地点
- 前端显示城市、地区和国家
- 用户选择目标地点
- 根据所选地点获取天气
- 前端不能直接调用 Open-Meteo

### FS3：AI 天气建议

增加一个 AI 相关的编程任务，但不需要真实的 AI API 或模型。

请使用两个后端实现中提供的模拟 AI 服务：

- Python/FastAPI
- Java/Spring Boot

该模拟服务应接收当前天气数据，并返回类似下面的结构：

```json
{
  "summary": "Warm and windy conditions.",
  "recommendation": "Stay hydrated when outdoors.",
  "riskLevel": "medium"
}
```

后端要求：

- 使用当前天气数据调用提供的模拟 AI 服务
- 将 AI 输出视为不可信数据
- 验证 summary 和 recommendation 为非空字符串
- 仅接受 low、medium、high 作为 riskLevel
- 处理格式错误、缺少字段、无效风险等级和服务失败
- 返回安全回退结果，同时不破坏正常天气响应
- 确保 AI 生成文本不会替换事实天气值

前端要求：

- 在一个简单区域中显示 AI 的 summary、recommendation 和 riskLevel
- 如果 AI 建议不可用，仍保留事实天气结果

不要实现候选人的最终 FS3 方案。只需添加可供候选人调用和集成的模拟 AI 服务起始代码。

模拟行为必须确定性且易于测试，而不是随机。请提供一个简单的可配置测试模式或方法，能够生成：

- 有效输出
- 格式错误输出
- 缺少字段
- 无效风险等级
- 超时或异常

请将模拟服务保持简短，并明确标记为面试脚手架。不要引入真实的 OpenAI 依赖、API 密钥或外部 AI 服务。
