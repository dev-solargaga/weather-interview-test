# 候选人任务：天气搜索应用

你将接手一个小型全栈天气应用。

目前，该应用已经可以搜索城市并显示基础的当前天气信息，但项目是在较短时间内完成的，因此在代码质量、用户体验、稳定性和可维护性方面仍有改进空间。

你的任务是将该应用完善为一个更完整、更可靠的天气搜索应用。

## 目标

实现一个天气页面，让用户可以搜索城市并查看：

- 系统解析后的城市和国家；
- 当前天气情况；
- 多日天气预报；
- 清晰的加载、空数据和错误状态。

前端只能调用本地后端，不能直接请求 Open-Meteo。

## 必做任务

### 1. 运行并检查项目

在本地运行前端和后端。

检查现有实现，并找出以下方面的问题：

- 代码质量；
- API 设计；
- 用户体验；
- 输入验证；
- 错误处理；
- 稳定性；
- 可维护性。

在开始修改之前，请先简要说明：

- 你认为当前项目的主要问题是什么；
- 你计划进行哪些改进；
- 为什么优先选择这些改进。

### 2. 改进城市搜索功能

用户必须能够通过城市名称进行搜索。

应用应当：

- 拒绝空白搜索；
- 自动移除多余空格；
- 正确处理不存在的城市；
- 请求进行中时避免重复提交；
- 显示清晰的加载状态；
- 请求失败时显示有意义的错误信息。

你也可以选择增加：

- 搜索防抖；
- 请求取消；
- 最近搜索记录；
- 搜索建议。

### 3. 显示当前天气

页面应显示以下内容：

- 系统解析后的城市名称；
- 国家；
- 当前温度；
- 体感温度；
- 风速；
- 天气代码或用户可读的天气描述；
- 日期和当地时间（如数据可用）。

即使部分天气字段缺失，页面仍应保持可用。

### 4. 显示多日天气预报

增加天气预报区域，至少显示未来五天的天气。

每一天应显示：

- 日期或星期；
- 最低温度；
- 最高温度；
- 天气代码或天气描述；
- 降水概率（如数据可用）；
- 风力信息（如数据可用）。

预报内容应以清晰、易读的方式呈现。

### 5. 改进后端

后端必须：

- 接收前端传入的城市名称；
- 使用 Open-Meteo 地理编码接口获取地点信息；
- 使用 Open-Meteo 获取当前天气和每日预报；
- 返回适合前端使用的整洁响应结构；
- 返回合理的 HTTP 状态码；
- 处理第三方 API 超时和错误；
- 避免将第三方 API 的原始错误直接暴露给用户。

你可以考虑将以下逻辑进行适当拆分：

- 路由或控制器逻辑；
- 天气服务逻辑；
- 响应模型；
- 第三方 API 请求逻辑。

本任务不要求进行完整的架构重写。

### 6. 验证

请至少验证以下情况：

1. 有效城市，例如 `Singapore`
2. 另一个不同国家的有效城市
3. 不存在的城市
4. 空白搜索
5. 包含多余空格的搜索内容
6. 模拟或真实的后端/API 请求失败
7. 短时间内连续执行多次搜索

请准备说明每种情况的处理结果。

## AI 工具使用说明

允许使用 AI 工具。

在要求 AI 修改代码之前，你必须先说明：

- 你对问题的理解；
- 你提出的解决方案；
- 你准备让 AI 协助处理的具体部分。

使用 AI 后，你仍需负责：

- 检查生成的代码；
- 测试所有修改；
- 验证实现是否正常运行；
- 理解并能够解释最终解决方案。

我们会根据实际情况公平评估 AI 的使用。候选人不会仅仅因为使用 AI 而被扣分。我们将重点评估 AI 是否被合理使用、候选人是否理解所提出的解决方案，以及是否能够自行检查、测试、验证并解释最终代码。
